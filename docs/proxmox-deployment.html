Code-changes review — Docker → direct LXC What (if anything) needs to change in the three repos
▶

Result: zero application code changes. All three apps already read their DB/Redis location from environment variables and default to localhost. The Docker setup was the detour — bare-metal was always supported.

Corporate (PUGHoldingWebSite)

  • backend/app/core/config.py — postgres_host defaults to localhost, redis_url to redis://localhost:6379/0. No change.
  • Frontend bakes NEXT_PUBLIC_API_BASE_URL at build time — set it in .env.production before npm run build.
  • Config-only: bind backend/frontend to 0.0.0.0 (the shipped systemd units bind 127.0.0.1 because nginx used to be on the same host; here nginx is a separate CT). Units below already do this.

Housing (Employee-Housing-Control-Portal)

  • backend/config.py resolves the DB URL as: DATABASE_URL → assembled POSTGRES_* → localhost fallback. No change.
  • gunicorn.conf.py already binds 0.0.0.0:5000. Entry points wsgi:app and celery -A celery_worker.celery are real.
  • Config-only: you author 4 systemd units (web, api, celery worker, celery beat) and install libmagic1 (python-magic). No code edits.

Finance (PUGFinanceApp)

  • App factory app:create_app(); PostgreSQL-only via DATABASE_URL; migrations flask --app "app:create_app()" db upgrade. No change.
  • On Linux use waitress-serve (already in requirements) or gunicorn instead of the Windows serve.py path. No code edit — just the service command.
  • No Node, no Redis, no Celery, no JS build — the simplest of the three.

Common to all: every backend targets Python 3.11 (Debian 12's native Python) and both frontends are Next.js 14 (Node 20 LTS). One golden CT template covers everything.

Install Proxmox VE on the server ISO → bootable USB → install → first login
▶

1.1 Create the installer

  • Download Proxmox VE ISO from proxmox.com/downloads.
  • Write it to a USB stick with balenaEtcher or Rufus (DD mode).
  • Boot the server from the USB (enable virtualization / VT-x/AMD-V in BIOS first).

1.2 Run the installer

  • Target disk: pick your system disk. Tip choose ZFS (RAID1) if you have two disks — you get snapshots + bit-rot protection. Otherwise ext4/LVM is fine.
  • Country / timezone / keyboard.
  • Set a strong root password + admin email.
  • Management network — set a static address:
    • Hostname: pve.parisunitedgroup.local
    • IP/CIDR: 192.168.100.10/24
    • Gateway: 192.168.100.1
    • DNS: 1.1.1.1
  • Finish, reboot, remove the USB.

1.3 First login

From any PC on the LAN open the web UI (accept the self-signed cert warning):

https://192.168.100.10:8006

Log in as root with the password you set. You can dismiss the "No valid subscription" notice — the next step switches to the free repo.

Open the host shell: Proxmox UI → select the node → Shell (or SSH to root@192.168.100.10). Run everything here as root.

2.1 Switch to the no-subscription repo & update

PVE 9 = Debian 13 "trixie" with the new deb822 .sources format; PVE 8 = Debian 12 "bookworm" with one‑line .list files. Check which you have, then use the matching block:

. /etc/os-release; echo "$VERSION_CODENAME"   # trixie = PVE 9, bookworm = PVE 8

PVE 9 (trixie):

# enterprise repos are .sources here — a sed on .list won't disable them
mv /etc/apt/sources.list.d/pve-enterprise.sources{,.disabled} 2>/dev/null
mv /etc/apt/sources.list.d/ceph.sources{,.disabled} 2>/dev/null
cat > /etc/apt/sources.list.d/pve-no-subscription.sources <<'EOF'
Types: deb
URIs: http://download.proxmox.com/debian/pve
Suites: trixie
Components: pve-no-subscription
Signed-By: /usr/share/keyrings/proxmox-archive-keyring.gpg
EOF
apt update && apt -y full-upgrade

PVE 8 (bookworm):

sed -i 's/^deb/#deb/' /etc/apt/sources.list.d/pve-enterprise.list 2>/dev/null
sed -i 's/^deb/#deb/' /etc/apt/sources.list.d/ceph.list 2>/dev/null
echo "deb http://download.proxmox.com/debian/pve bookworm pve-no-subscription" > /etc/apt/sources.list.d/pve-no-subscription.list
apt update && apt -y full-upgrade

2.2 Useful host tools

Keep the hypervisor lean — only operational tools on the host. App runtimes (Node, Python venvs, Postgres) go INSIDE the containers, not here.

apt install -y git curl vim htop tmux ca-certificates ufw

The default vmbr0 bridges your physical NIC to the LAN — that is what the containers attach to. Verify it carries your static IP:

ip -br a show vmbr0
cat /etc/network/interfaces   # vmbr0 should have address 192.168.100.10/24, gateway 192.168.100.1

CT 110 — Cloudflare Tunnel Dedicated LXC running cloudflared → https://192.168.100.50:443
▶

3.1 Create the container (run on the Proxmox host)

pct create 110 local:vztmpl/debian-12-standard_12.7-1_amd64.tar.zst \
  --hostname cf-tunnel \
  --cores 1 --memory 512 --swap 512 \
  --rootfs local-lvm:4 \
  --net0 name=eth0,bridge=vmbr0,ip=192.168.100.49/24,gw=192.168.100.1 \
  --nameserver 1.1.1.1 \
  --unprivileged 1 --features nesting=1 --onboot 1 \
  --password CHANGE_ME --start 1
pct enter 110

3.2 Install cloudflared (inside CT 110)

apt update && apt install -y curl
# install from Cloudflare's apt repo (lands in /usr/bin, always on PATH, updatable)
mkdir -p --mode=0755 /usr/share/keyrings
curl -fsSL https://pkg.cloudflare.com/cloudflare-main.gpg | tee /usr/share/keyrings/cloudflare-main.gpg >/dev/null
echo 'deb [signed-by=/usr/share/keyrings/cloudflare-main.gpg] https://pkg.cloudflare.com/cloudflared bookworm main' > /etc/apt/sources.list.d/cloudflared.list
apt update && apt install -y cloudflared
mkdir -p /etc/cloudflared
cloudflared --version

(The manual binary download works too, but if /usr/local/bin isn't on $PATH you get cloudflared: command not found — the apt package avoids that.)

3.3 Point the tunnel at the nginx CT (the key change)

You currently send traffic to localhost:443. Re-point every hostname to the real nginx CT IP 192.168.100.50:443. Two ways depending on how your tunnel is managed:

A) Remotely-managed tunnel (Cloudflare dashboard → Zero Trust → Networks → Tunnels): run it with the token, then edit each Public Hostname in the dashboard.

# Install as a service using the token Cloudflare shows for your tunnel:
cloudflared service install TUNNEL_TOKEN_HERE
systemctl enable --now cloudflared
systemctl status cloudflared --no-pager

Then in the dashboard, for each hostname set Service = HTTPS, URL = 192.168.100.50:443, and under Additional settings → TLS enable No TLS Verify (origin cert is a Cloudflare Origin CA cert). Hostnames: accommodation, pugfin, parisunitedgroup.com, www.parisunitedgroup.com.

B) Locally-managed tunnel (config file). Create the config:

/etc/cloudflared/config.yml
tunnel: TUNNEL_ID_HERE
credentials-file: /etc/cloudflared/TUNNEL_ID_HERE.json

ingress:
  - hostname: accommodation.parisunitedgroup.com
    service: https://192.168.100.50:443
    originRequest:
      noTLSVerify: true
  - hostname: pugfin.parisunitedgroup.com
    service: https://192.168.100.50:443
    originRequest:
      noTLSVerify: true
  - hostname: parisunitedgroup.com
    service: https://192.168.100.50:443
    originRequest:
      noTLSVerify: true
  - hostname: www.parisunitedgroup.com
    service: https://192.168.100.50:443
    originRequest:
      noTLSVerify: true
  - service: http_status:404

Authenticate, copy your tunnel credentials JSON into /etc/cloudflared/, route DNS, then install the service:

# If creating fresh (skip if you already have the tunnel + json):
cloudflared tunnel login
cloudflared tunnel create pug-edge          # prints the TUNNEL_ID + writes the json

# Map each hostname to this tunnel (CNAME in Cloudflare DNS):
cloudflared tunnel route dns pug-edge accommodation.parisunitedgroup.com
cloudflared tunnel route dns pug-edge pugfin.parisunitedgroup.com
cloudflared tunnel route dns pug-edge parisunitedgroup.com
cloudflared tunnel route dns pug-edge www.parisunitedgroup.com

cloudflared --config /etc/cloudflared/config.yml service install
systemctl enable --now cloudflared
systemctl status cloudflared --no-pager

cloudflared dials out to Cloudflare on 443 — there is no router port-forwarding anywhere, exactly as you wanted. Type exit to leave the container.

4.5 The site config — pull it from the repo

Don't hand‑paste this. The repo ships a ready LXC build at lxc/nginx.conf: static CT‑IP upstreams (so one down app can't stop nginx booting) and the nginx‑1.22 listen … ssl http2 form — Debian 12's nginx is 1.22, where http2 on; errors with unknown directive.

apt install -y git
git clone https://github.com/Shamseer1988/edge-proxy.git /opt/edge-proxy
cp /opt/edge-proxy/lxc/nginx.conf /etc/nginx/conf.d/default.conf
cp /opt/edge-proxy/snippets/*    /etc/nginx/snippets/
# update later:  cd /opt/edge-proxy && git pull  then re-copy the two paths + reload nginx

Edit the upstream CT IPs at the top of /etc/nginx/conf.d/default.conf if yours differ from 192.168.100.51‑53. The full config is shown below for reference (identical to lxc/nginx.conf):

/etc/nginx/conf.d/default.conf (reference — same as lxc/nginx.conf)
# Edge proxy on 192.168.100.50. TLS terminates here; Cloudflare Tunnel
# sends every hostname to https://192.168.100.50:443. Real client IP
# arrives via CF-Connecting-IP from the tunnel CT (192.168.100.49).
set_real_ip_from 192.168.100.0/24;
real_ip_header CF-Connecting-IP;
real_ip_recursive on;

ssl_session_cache shared:SSL:50m;
ssl_session_timeout 1d;
ssl_session_tickets off;
# NOTE: no `gzip on;` — Debian's main nginx.conf already sets it (duplicate = emerg).
gzip_types text/plain application/json application/javascript text/css image/svg+xml application/xml;
gzip_min_length 1024;

# ---- Upstreams: the app LXC containers ----
upstream pugweb_frontend  { server 192.168.100.51:3000; keepalive 32; }
upstream pugweb_api       { server 192.168.100.51:8000; keepalive 32; }
upstream housing_frontend { server 192.168.100.52:3000; keepalive 32; }
upstream housing_backend  { server 192.168.100.52:5000; keepalive 32; }
upstream pugfin_app       { server 192.168.100.53:5000; keepalive 32; }

# ---- 80 -> 443 ----
server {
    listen 80 default_server;
    server_name accommodation.parisunitedgroup.com pugfin.parisunitedgroup.com parisunitedgroup.com www.parisunitedgroup.com;
    return 301 https://$host$request_uri;
}

# ---- default 443: drop unknown SNI ----
server {
    listen 443 ssl http2 default_server;
    listen [::]:443 ssl http2 default_server;
    server_name _;
    include /etc/nginx/snippets/ssl-common.conf;
    return 444;
}

# ---- Host 1: accommodation -> housing ----
server {
    listen 443 ssl http2;
    listen [::]:443 ssl http2;
    server_name accommodation.parisunitedgroup.com;
    include /etc/nginx/snippets/ssl-common.conf;
    include /etc/nginx/snippets/security-headers.conf;
    client_max_body_size 30M;

    location = /health { proxy_pass http://housing_backend/api/v1/health; access_log off; }
    location /api/ {
        include /etc/nginx/snippets/proxy-common.conf;
        proxy_buffering off; proxy_cache off; proxy_read_timeout 1h;
        proxy_pass http://housing_backend;
    }
    location / {
        include /etc/nginx/snippets/proxy-common.conf;
        proxy_pass http://housing_frontend; proxy_read_timeout 60s;
    }
}

# ---- Host 2: pugfin -> finance (single Flask app, serves its own static) ----
server {
    listen 443 ssl http2;
    listen [::]:443 ssl http2;
    server_name pugfin.parisunitedgroup.com;
    include /etc/nginx/snippets/ssl-common.conf;
    include /etc/nginx/snippets/security-headers.conf;
    client_max_body_size 30M;

    location = /health { access_log off; default_type text/plain; return 200 "ok\n"; }
    location / {
        include /etc/nginx/snippets/proxy-common.conf;
        proxy_pass http://pugfin_app; proxy_read_timeout 120s;
    }
}

# ---- Host 3: corporate -> pugweb (the app sets its own CSP, so none here) ----
server {
    listen 443 ssl http2;
    listen [::]:443 ssl http2;
    server_name parisunitedgroup.com www.parisunitedgroup.com;
    include /etc/nginx/snippets/ssl-common.conf;
    include /etc/nginx/snippets/security-headers.conf;
    client_max_body_size 20M;

    if ($host = www.parisunitedgroup.com) { return 301 https://parisunitedgroup.com$request_uri; }

    location = /health { access_log off; default_type text/plain; return 200 "ok\n"; }
    location /api/v1/ {
        include /etc/nginx/snippets/proxy-common.conf;
        proxy_buffering off; proxy_cache off; proxy_read_timeout 1h;
        proxy_pass http://pugweb_api;
    }
    location /_next/static/ {
        include /etc/nginx/snippets/proxy-common.conf;
        proxy_pass http://pugweb_frontend;
        expires 30d; add_header Cache-Control "public, max-age=2592000, immutable";
    }
    location / {
        include /etc/nginx/snippets/proxy-common.conf;
        proxy_pass http://pugweb_frontend; proxy_read_timeout 60s;
    }
}

8.8 Verify — mind the local‑test SNI gotcha

# use --resolve so SNI matches the host (a DIFFERENT cert lives on this domain)
curl -ksS --resolve teekeyhospitality.com:443:127.0.0.1 -I https://teekeyhospitality.com/ | head -1                # 200 frontend
curl -ksS --resolve www.teekeyhospitality.com:443:127.0.0.1 -I https://www.teekeyhospitality.com/ | head -1        # 301
curl -ksS --resolve zeroone.teekeyhospitality.com:443:127.0.0.1 -I https://zeroone.teekeyhospitality.com/api/health | head -1   # 200 API

Don't test with -H "Host:" https://127.0.0.1 on this domain — it has a different cert from the PUG vhosts, so SNI (no SNI / 127.0.0.1 → default cert) vs Host mismatch makes nginx reject HTTP/2 with PROTOCOL_ERROR. Always use --resolve. Also normal: {"error":"Not found"} at the API root — its routes live under /api/.

The SPA calls the API cross‑origin at this same public origin, so CORS_ORIGINS must list https://legal.parisunitedgroup.com. Quote any value containing spaces (APP_NAME, BRAND_COMPANY_NAME) — the manual . ./.env sourcing in 9.5 word-splits unquoted spaces (the shell prints command not found; DATABASE_URL/SEED_ADMIN_* have none, so migrate + seed still run), and systemd's EnvironmentFile honors the quotes too. Keep APP_SECRET_KEY a space-free hex string.

/opt/pug-legal/frontend/.env.production (baked into the build)

NEXT_PUBLIC_* is a build‑time constant — compiled into the JS at npm run build, so it must be set before building, not just in the service env. The app appends /api/v1/... itself, so the value is the bare origin (no /api suffix).

Next build fails on lint? Next.js runs ESLint inside next build and treats rule violations (e.g. react/no-unescaped-entities — a literal " in JSX) as hard errors that abort with Failed to compile and produce no build. Rebuild with npm run build -- --no-lint, or set eslint: { ignoreDuringBuilds: true } in the app's next.config.js and commit it upstream so redeploys keep building.

9.6 systemd units — copy the shipped ones, apply the topology‑B edits

The repo ships pug-backend.service + pug-frontend.service, but they bind 127.0.0.1 (they assume nginx on the same host). Here the edge nginx is a separate CT, so both must bind 0.0.0.0:

cp /opt/pug-legal/infra/systemd/pug-backend.service \
   /opt/pug-legal/infra/systemd/pug-frontend.service  /etc/systemd/system/

# backend: bind 0.0.0.0, ONE worker, trust the edge CT for forwarded headers
sed -i -e 's#--host 127.0.0.1#--host 0.0.0.0#' \
       -e 's#--workers 2#--workers 1#' \
       -e 's#--forwarded-allow-ips=127.0.0.1#--forwarded-allow-ips=192.168.100.50#' \
       /etc/systemd/system/pug-backend.service
# frontend: bind 0.0.0.0
sed -i 's#npm start -- -p 3000#npm start -- -H 0.0.0.0 -p 3000#' \
       /etc/systemd/system/pug-frontend.service

systemctl daemon-reload
systemctl enable --now pug-backend pug-frontend
systemctl status pug-backend pug-frontend --no-pager | head

Use ONE uvicorn worker. The backend runs its scheduler (scheduled reports, email‑queue drain, SLA + hearing reminders) in‑process via APScheduler. With --workers 2 the scheduler starts in each worker and every timed job double‑fires. Keep --workers 1 (or split a dedicated scheduler process) on this single‑CT deploy.
